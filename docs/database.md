# Database

FastRep currently uses PostgreSQL through Prisma. The schema contains only `User` and `RefreshToken`; there are no report or media tables yet.

## User

| Field           | Type             | Description                                                                                                |
| --------------- | ---------------- | ---------------------------------------------------------------------------------------------------------- |
| `id`            | `String`         | UUID primary key generated by Prisma.                                                                      |
| `fullName`      | `String`         | User's display name.                                                                                       |
| `email`         | `String`         | Unique normalized email used for authentication.                                                           |
| `passwordHash`  | `String`         | Argon2 password hash. A raw password is never stored.                                                      |
| `language`      | `String`         | Application language; database default is `en`. The current registration API accepts `en`, `uk`, and `ru`. |
| `photoUrl`      | `String?`        | Optional profile photo URL.                                                                                |
| `isPremium`     | `Boolean`        | Premium flag, defaulting to `false`. No subscription feature is currently implemented.                     |
| `createdAt`     | `DateTime`       | Creation timestamp, defaulting to the database insertion time.                                             |
| `updatedAt`     | `DateTime`       | Timestamp automatically updated by Prisma on changes.                                                      |
| `refreshTokens` | `RefreshToken[]` | One-to-many relation to the user's active refresh-token records.                                           |

Passwords are one-way hashed with Argon2 before insertion. Authentication verifies a submitted password against `passwordHash`; the hash is excluded from public user responses.

## RefreshToken

| Field       | Type       | Description                                                                     |
| ----------- | ---------- | ------------------------------------------------------------------------------- |
| `id`        | `String`   | UUID primary key. It is also used as the refresh JWT's `jti` identifier.        |
| `userId`    | `String`   | Foreign key referencing `User.id`; indexed for user-based lookup.               |
| `tokenHash` | `String`   | Argon2 hash of the issued refresh token. The raw refresh token is never stored. |
| `expiresAt` | `DateTime` | Server-side expiration timestamp for the token record.                          |
| `createdAt` | `DateTime` | Creation timestamp, defaulting to the insertion time.                           |
| `user`      | `User`     | Required relation to the owning user.                                           |

One user can have multiple refresh-token records because each login creates a new token and separate mobile sessions can coexist. Refresh tokens are stored as hashes so a database read does not reveal usable session credentials. Refresh replaces the current record with a new record, and logout deletes the supplied token's record.

The relation uses `onDelete: Cascade`: deleting a user automatically deletes all refresh-token records owned by that user.

## Schema changes and migrations

Edit `prisma/schema.prisma`, then create a migration with:

```bash
npm run prisma:migrate
```

Commit both the schema change and generated migration under `prisma/migrations`. Regenerate Prisma Client with `npm run prisma:generate`. Never edit generated files under `generated/prisma` manually.

## Planned models

Report and media models are planned but not implemented. Their fields, relations, storage lifecycle, and migrations will be designed in a separate feature.
